# 项目整改升级全案执行计划

## 1. 目标

### P0

恢复项目可运行、可构建、可部署。

### P1

统一内容源、类型契约、视觉 token 与页面主链，结束新旧实现并行失控状态。

### P2

围绕“中国古代建筑的文化内核与技术智慧”重建叙事，把艺术表达与科技展示绑定到真实内容上。

### P3

为后续扩展新建筑、新模块和新交互留出清晰工程边界。

## 2. 本轮执行策略

### 2.1 内容与叙事

- 选取 5 个代表性案例作为首批稳定内容集：
  - 太和殿
  - 祈年殿
  - 布达拉宫
  - 留园
  - 福建土楼
- 每个案例统一使用 5 个主题模块：
  - 空间秩序
  - 结构逻辑
  - 营造工艺
  - 装饰象征
  - 保护传承

### 2.2 视觉与组件

- 保留深色底 + 金石色强调的视觉方向。
- 以 `site/*` 组件作为新主链展示层：
  - `BuildingCard`
  - `ModuleVisualizer`
  - `MonumentIllustration`
  - `SectionHeading`
- 统一全局样式入口为 `src/styles/index.css`，并显式引入：
  - `fonts.css`
  - `tailwind.css`
  - `theme.css`

### 2.3 交互与状态

- 采用“首页总览 + 详情页深读”的双层导航结构。
- 模块进度使用 `buildingId:moduleId` 作为唯一键，避免跨案例串联。
- 为每个案例完成状态提供专题藏印反馈。

### 2.4 技术架构

- 单一内容源：`src/app/data/buildings.ts`
- 单一类型源：`src/app/types/index.ts`
- 单一进度上下文：`src/app/context/ProgressContext.tsx`
- 补充 `tsconfig.json` 与 `pnpm typecheck`

## 3. 已执行节点

### 第一阶段：诊断

- 完成全项目结构扫描
- 完成构建阻塞定位
- 完成内容源与类型分叉定位

### 第二阶段：重构

- 重建主页面链路
- 重建数据模型与文化内容结构
- 重建核心展示组件
- 重建进度状态逻辑

### 第三阶段：治理

- 修复入口与错误页乱码
- 收拢样式 token
- 新增类型检查能力

### 第四阶段：验收

- `pnpm typecheck` 通过
- `pnpm build` 通过

## 4. 验收标准

### 已满足

- 项目可正常构建
- 主页面与详情页可正常编译
- 活跃主链无已知乱码
- 数据结构统一
- 进度状态可稳定追踪

### 下一阶段建议标准

- 完成历史文件归档或删除
- 完成依赖瘦身
- 完成更细颗粒度的代码拆包与懒加载
- 补充视觉截图验收与跨终端人工回归

## 5. 协同规则

1. 后续新增案例时，只能从 `src/app/data/buildings.ts` 增补，禁止在页面层硬编码正文。
2. 后续新增样式变量时，只能进入 `theme.css` 与 `fonts.css`，禁止再造第四套 token。
3. 后续新增交互组件时，优先落在 `src/app/components/site`，页面只负责编排。
4. 新增重大方向前，应先确认：
   - 是否扩展现有 5 模块
   - 是否引入新建筑类型
   - 是否需要更换视觉基调或技术栈
